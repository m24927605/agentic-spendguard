**Decision**

Fix anyway before closing POST_GA_05. This P2 is valid production-readiness debt, not out-of-scope. The final arbitration is: make the targeted Bedrock SDK fix, then close without another review round.

**Rationale**

The code already models and persists `ProviderCount.request_id`, and the HTTP-compatible Llama backend already captures the provider request ID. Dropping the Bedrock SDK request ID creates an avoidable observability gap for production drift samples and alert CloudEvents. The fix is narrow, low-risk, and directly aligned with provider coverage expectations.

- After `count_tokens().send().await`, populate:

```rust
let request_id = output.request_id().map(str::to_owned);
```

- Pass that value into `ProviderCount { request_id, ... }` instead of `None`.
- Do not change schema, event contracts, or HTTP-compatible behavior.
- Add or update a focused test proving the Bedrock SDK success path preserves a request ID into `ProviderCount`.
- Run the tokenizer test target covering Llama provider clients, plus formatting/lint checks normally required for this repo.

Low. The main risk is testability around constructing an AWS SDK output with request metadata; if that is awkward, isolate the extraction behind a small helper and test that helper plus the Bedrock mapping path. Runtime behavior should remain unchanged except for improved correlation metadata.

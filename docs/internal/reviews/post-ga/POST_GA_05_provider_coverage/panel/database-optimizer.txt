**Decision**
Fix anyway before POST_GA_05 ships. This P2 is production-readiness relevant, not out-of-scope.

**Rationale**
The Bedrock SDK path is a first-class production provider path. Dropping `x-amzn-requestid` creates asymmetric observability versus the HTTP-compatible Llama backend and breaks correlation between drift samples, alert CloudEvents, and AWS Bedrock logs. The fix is low-risk and localized: preserve `CountTokensOutput::request_id()` into `ProviderCount.request_id`.

**Required Implementation/Tests**
- In `services/tokenizer/src/shadow/provider_clients/llama.rs`, import the SDK request ID trait:
  `use aws_types::request_id::RequestId;`
- After `count_tokens().send().await`, capture:
  `let request_id = output.request_id().map(str::to_owned);`
- Store that value in the returned `ProviderCount { request_id, ... }`.
- Preserve existing `None` behavior when the SDK response has no request ID.
- Add/extend a Bedrock Llama unit test or mocked-client test proving:
  - SDK request ID becomes `ProviderCount.request_id`.
  - Missing SDK request ID remains `None`.
  - HTTP-compatible provider-request-id behavior is unchanged.
- Run the tokenizer/provider test target covering Llama Bedrock shadow counting.

**Residual Risk**
Low. Main risk is dependency/import friction around `aws_types`; if it is only transitive today, add it explicitly to the tokenizer crate. No schema or persistence changes should be needed because `ProviderCount.request_id` is already persisted.

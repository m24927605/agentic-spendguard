**Decision**

Fix anyway before POST_GA_05 is accepted. This is a valid remaining P2, not out-of-scope, and no waiver is appropriate.

**Rationale**

The Bedrock SDK path drops a provider correlation identifier that the HTTP-compatible path already preserves. Since `ProviderCount.request_id` is already part of persisted evidence, this is a narrow parity and production-readiness fix, not a schema or scope expansion. Losing `x-amzn-requestid` weakens incident investigation and alert-to-provider-log correlation for Bedrock-backed Llama drift samples.

**Required implementation/tests**

- In `services/tokenizer/src/shadow/provider_clients/llama.rs`, import/use `aws_types::request_id::RequestId`.
- After `client.count_tokens().send().await`, extract:

```rust
let request_id = output.request_id().map(str::to_owned);
```

- Store that value in the Bedrock `ProviderCount.request_id` instead of `None`.
- Add/adjust a focused test proving the Bedrock success path propagates a request id into `ProviderCount`.
- Existing HTTP-compatible request-id behavior must remain unchanged.
- Acceptance checks:
  - Targeted tokenizer/provider client tests pass.
  - Any existing tokenizer shadow/persistence tests pass.
  - A test fixture or mock verifies `Some("...")` from `CountTokensOutput::request_id()` reaches persisted sample evidence.

**Residual risk**

AWS SDK may still return `None` for some responses or errors, so correlation is best-effort. This fix closes the avoidable success-path gap; error-path request-id capture can be handled separately if current telemetry does not already preserve it.

### `ledger` — Main reserve path (post_ledger_transaction) never enforces non-negative available_budget — no BUDGET_EXHAUSTED hard cap, enabling over-spend

- **Location:** `services/ledger/migrations/0012_post_ledger_transaction.sql:379`
- **Category:** data-integrity
- **Impact:** The ledger is documented as the single source of truth and hard cap, but the primary ReserveSet path will happily debit available_budget below zero. The only thing preventing over-spend is the sidecar's QueryBudgetState read before reserving. Any sidecar bug, projection lag, or two reservations that each individually fit but jointly exceed the budget will silently over-reserve and over-commit real money. This is the cardinal fail-open: a DENY (budget exhausted) becomes an ALLOW. Serializable isolation does not save it — there is no predicate to violate; both writes are accepted.
- **Proposed fix:** After locking the available_budget account row (step 6) and before/at the balance assertion, compute the post-write available_budget balance for each touched (budget,window,unit) and RAISE 'BUDGET_EXHAUSTED' (USING ERRCODE mapped to DomainError::BudgetExhausted) when it would go negative — mirroring post_session_reserve's session_account_balance check. Make the ledger, not the sidecar, the authority for the hard cap.
- **Verifier note:** Correct in direction and authority placement (enforce in the ledger proc, not the sidecar), and it correctly mirrors the existing session-path check. It is safe and near-minimal with these caveats the implementer MUST honor: (a) Orientation — available_budget is credit-positive (query_budget_state.rs:126: balance = SUM(credit)-SUM(debit)); the post-write balance must use this orientation, not raw net_debit, or the sign inverts. (b) Cumulative, not per-tx — compute the balance from ALL committed entries for each touched (budget,window,unit) available_budget account, because prior reservations are the source of over-spend; checking only this tx's entries would miss it. (c) Concurrency is already handled — step 6 (0012:263-275) locks the exact available_budget rows FOR UPDATE, so reading the post-write balance under that held lock is the correct serialization; this is why the row-lock approach beats relying on serializable predicate detection (the finding rightly notes there is currently no predicate to violate). (d) Multi-op safety — the proc also serves release/refund/control-plane ops which only CREDIT available_budget (increase balance), so a 'balance ≥ 0 after write' floor won't wrongly fire on them; scope the RAISE to accounts whose net effect is a debit (reserve). (e) Error mapping — RAISE with a distinct ERRCODE and add a BudgetExhausted arm in reserve_set.rs's error match so it surfaces as the Outcome::Error proto business-DENY variant, NOT a gRPC transport error (retriable). (f) Add a main-path regression test (two reserves individually fitting but jointly exceeding) plus a demo mode, per demo-as-quality-gate. OPEN PREREQUISITE the fix description understates: it assumes available_budget is pre-funded with the budget ceiling. The implementer must first confirm the budget-grant funding entry exists (credits available_budget to the limit); if budgets are NOT funded into the ledger account, a naive 'balance ≥ 0' floor would fire on the very first reserve and break all traffic, and the fix would need to source the ceiling (e.g., from budget config) rather than the account balance. query_budget_state and the session model strongly imply funding exists, but verify before shipping. Do NOT move the cap into the sidecar — that would re-introduce the same TOCTOU; the ledger must be the authority.

### `plugins` — Kong Go body_filter treats full buffered body as a chunk stream; end-of-body commit never fires on real Kong → reservations leak un-committed (uncounted spend)

- **Location:** `plugins/kong/spendguard-go/body_filter.go:118`
- **Category:** data-integrity
- **Impact:** On a real Kong deployment the commit/release (POST /v1/trace) never fires: every ALLOW'd call holds its reservation until TTL sweep and the realized spend is never committed to the ledger. Budgets silently under-count actual usage — a money leak and a defeated cost control. If body_filter happens to fire twice the buffer doubles, the JSON parse fails, and a successful call is released as REJECTED instead — also wrong.

Read the full body once on body_filter (it is already buffered), parse usage, and emit the trace immediately, guarding with the CtxKeyCommitted dedup flag. Replace the isFinalChunk/empty-terminator accumulation with a single-shot read, and add an integration test against a real (or PDK-test-harness) Kong that exercises get_raw_body semantics rather than a synthetic chunk queue.
- **Verifier note:** Directionally correct and fail closed aligned. Do a single buffered read, parse it, and send the trace immediately, guarded by the existing dedup flag, and delete the dead accumulation code. Also handle the temp file path body branch and keep empty or unparseable bodies mapping to a release outcome. Replace the mock chunk queue test with a real Kong integration test.

- **Impact:** The Botpress->sidecar decision channel is unauthenticated and unencrypted-capable. ### `sidecar` — Subscription-meter cap is fully bypassable: hard-cap config + routing flag taken from untrusted client metadata with no gate and no server-side accumulation

- **Location:** `services/sidecar/src/subscription_meter/route.rs:97`
- **Category:** auth-trust
- **Impact:** An attacker (or any normal caller) on the subscription-meter lane can spend unbounded amounts on the flat-fee plan: omit `hard_cap_at_atomic` (→ no cap), or send `current_consumed_atomic=0` each call (→ cap only fires if a single request alone exceeds it). Setting `reservation_source=SUBSCRIPTION_METER` also makes the sidecar NEVER open a ledger transaction (no fencing-backed budget oracle), so the meter flag is a general ledger-bypass switch. This silently defeats the spend control and is decided as CONTINUE in production.
- **Proposed fix:** Do not trust client-supplied routing or cap config for an enforcement decision. Gate the entire meter lane behind an explicit server-side opt-in (mirroring `allow_untrusted_budget_metadata`) until the ledger-backed `subscription_meters` lookup lands; when not gated, fail closed (DENY) rather than CONTINUE. Read `current_consumed_atomic` and `hard_cap_at_atomic` from a signed/fenced server-side source (ledger), never from `runtime_metadata`. Treat a missing hard cap on the meter lane as fail-closed, not unlimited.
- **Verifier note:** Correct, safe, and consistent with the existing pattern. The right move is to gate the meter lane behind an explicit server-side opt-in mirroring allow_untrusted_budget_metadata (default off in production), and to never source current_consumed/hard_cap from runtime_metadata for an enforcement decision — read them from the fenced ledger subscription_meters source (already acknowledged as the post-D13 slice). Treat a missing hard cap on the meter lane as fail-closed, not unlimited. One refinement on minimality: failing closed unconditionally the instant a SUBSCRIPTION_METER request arrives would break the legitimate flat-fee passthrough whose purpose is to avoid double-counting against the vendor flat plan; so the minimal safe interim is — when the server-side flag is off, reject SUBSCRIPTION_METER as unconfigured (DENY / FailedPrecondition) or route through the normal ledger path, rather than silently honoring client-supplied caps. The proposed fix captures this intent correctly.

### `sidecar` — UDS adapter server performs NO SO_PEERCRED peer-uid check and hardens no socket permissions despite documenting SO_PEERCRED auth

- **Location:** `services/sidecar/src/main.rs:333`
- **Category:** auth-trust
- **Impact:** Any local process that can reach the UDS path can call RequestDecision (mint ALLOW/DENY + ledger reservations), ReleaseReservation (release budget), and ConfirmPublishOutcome/EmitTraceEvents (drive commits/releases) impersonating the trusted in-pod adapter. In a shared/compromised container this is a full bypass of the trust boundary the product relies on. The doc comment makes operators believe this control exists when it does not.
- **Proposed fix:** Implement the documented check: on each accepted `UnixStream`, read peer credentials (e.g. `stream.peer_cred()` / SO_PEERCRED) and reject connections whose uid != the configured expected app uid before handing the stream to tonic. Additionally `fs::set_permissions(&uds_path, 0o600)` (or 0o660 with a shared gid) after bind. If the check is deferred, change the docstring so it does not claim an enforcement that is absent.
- **Verifier note:** Proposed fix is correct, safe, minimal, and fail-closed, with two refinements. (a) Permission hardening is the universal must-do and exactly mirrors existing tokenizer code: after `UnixListener::bind` in main.rs, call `std::fs::set_permissions(&uds_path, std::fs::Permissions::from_mode(0o600))` (or 0o660 with a shared gid matching the chart fsGroup 65532) — do it after bind and before the listener accepts, and fail-closed (propagate the error with `?`/context so a chmod failure aborts startup rather than serving a world-accessible socket). (b) Peer-uid enforcement: in the `incoming` stream, call `stream.peer_cred()` (tokio UnixStream returns UCred; use `.uid()`) and only `yield Ok(stream)` when uid matches a configured expected app uid; on missing/mismatched cred drop the connection (do not yield) — this is fail-closed. This requires a small additive config field (e.g. `expected_app_uid`, currently absent from config.rs); when unset, default to the process's own uid (getuid) rather than fail-open. Scope caveat: SO_PEERCRED only applies to the in-pod/same-uid adapter path; for the sibling-pod/hostPath production shape the design (D01 §3.3, D09) already mandates mTLS+SPIFFE instead of peer-cred, so the uid check should guard the local UDS adapter path specifically, while the permission hardening applies everywhere. If the team intends to defer the uid check, the adapter_uds.rs:1-6 docstring and sidecar-architecture-spec-v1alpha1.md:223-229 must be amended to stop claiming an enforced control, but deferral is not recommended given the node-wide hostPath exposure.

- **Category:** auth-trust
- **Impact:** Any workload with a CA-valid client cert can submit audit events claiming an arbitrary producer_id and tenant_id. Defense-in-depth (per-tenant SVID work in HARDEN_08) is not enforced on the ingest path; trust collapses to 'has a cert from our CA'.

Not fully minimal/complete as written: the more fundamental gap is the verifier's missing key->producer/tenant authorization binding — a perfect SAN check is still bypassed by anyone presenting a CA-valid cert who also holds a registered key, so the real fix should also extend the trust store to a key_id -> {producer_id, allowed_tenants} map and assert it in verify_or_handle. The fix MUST fail-closed: when mTLS is configured, reject Unauthenticated on missing/unparseable peer cert SAN, never admit. It must not break existing in-cluster forwarders whose certs may not yet carry SPIFFE URI SANs, so this needs a rollout/migration plan (issue SANs to all producers first), not a one-line patch. ### `crates` — verify-chain --check-prediction-mirror (default ON) never compares anything and always exits 0 — silent fail-open audit gate

- **Location:** `services/canonical_ingest/src/bin/verify_chain.rs:222`
- **Category:** fail-open
- **Impact:** An operator running verify-chain (the audit-integrity tool) with the default --check-prediction-mirror=true gets a green exit code regardless of whether the on-disk audit_outbox prediction columns actually match the signed CloudEvent proto bytes. A producer that writes a mismatched mirror (e.g. NULL column vs nonzero proto sentinel, or a tampered row) would pass verification undetected. This defeats the documented §11.3 integrity check on the audit chain.

- **Verifier note:** Correct in direction; the minimal variant is the right fail-closed choice. Implementing the full per-row Postgres cross-check is the proper end state but is NOT minimal (needs a live ledger DB + verifier/key-registry wiring, the same dependency the library punted on). The safe, minimal fix is the second alternative: make --check-prediction-mirror=true hard-fail with a non-zero exit (round-3 M5 used exit 2) until the real scan ships, so the default gate is fail-closed instead of green. ### `egress_proxy` — Streaming success without a parseable usage event releases the reservation, leaking real spend off-budget

- **Location:** `services/egress_proxy/src/forward.rs:1222`
- **Category:** data-integrity
- **Impact:** Silent budget under-count / spend leak on the streaming happy path. Agents that stream (the SDK default) can consume tokens that never hit the budget whenever usage parsing yields None on an otherwise-successful stream, defeating the cost guardrail. This is fail-open on accounting, the cardinal failure mode for this product.
- **Proposed fix:** On a stream that ended WITHOUT a stream error (stream_errored == false) but with no usage, do NOT release as ProviderError. Either commit the conservative reservation estimate (reserved_tokens) as a floor -- mirroring the non-streaming path's parse_usage_tokens(...).unwrap_or(inputs.estimated_tokens) at forward.rs:821 -- or emit a distinct SUCCESS-but-usage-missing outcome and commit the estimate. Reserve the release path for true stream errors only.
- **Verifier note:** Proposed fix is correct, safe, and minimal. Thread the stream_errored boolean from parse_usage_from_sse_stream to the commit task (replace the oneshot's Option<i64> payload with a small enum/Result: e.g. UsageOutcome::Parsed(n) / Closed(no usage, no error) / Errored). On Closed-without-usage, commit the conservative reservation estimate (reserved_tokens / inputs.estimated_tokens) as a floor, mirroring the existing non-streaming behavior at forward.rs:821, so accounting stays fail-closed. Reserve release_on_upstream_error for true stream errors (the Errored case) only. This adds no double-billing risk — the reservation TTL still backstops genuine proxy crashes, and committing a floor on a successful token-consuming call is strictly safer than zero-debiting it. Note the conflated Ok(None) branch is also encoded in the spec (egress-proxy-v0.2-streaming-sse.md:208-209), so the spec should be amended alongside the code to split success-no-usage from stream-error; emitting a distinct SUCCESS-but-usage-missing outcome/metric (as the fix suggests) is worth keeping for observability.

### `identity_policy` — K8sLease leader takeover is non-atomic (no resourceVersion CAS) → split-brain dual leaders

- **Location:** `services/leases/src/lib.rs:405`
- **Category:** concurrency
- **Impact:** When leaderElection.mode=k8s (a wired production mode in outbox_forwarder/ttl_sweeper/retention_sweeper main.rs), two pods can simultaneously hold the lease. The single-leader invariant is the only thing preventing concurrent singleton work; defeating it breaks leader election for audit forwarding, TTL sweeping, and retention — directly enabling the duplicate-forward data-integrity break below.

Patch::Apply with a stable field manager (force=false) or a resourceVersion-precondition merge makes takeover atomic; a concurrent loser gets 409 Conflict, which must map to Standby/retry — this preserves fail-closed (no work done) rather than fail-open dual leadership. Two refinements: (a) apply the same precondition/field-manager to the renew branch (line 385) and the release clear (line 462) for consistency, though the takeover branch at 416 is the split-brain-critical one; (b) ensure 409 maps to Standby/retry and never to a spurious Leader. The proposed two-pod kind integration test mirroring integration_postgres.rs is the right regression guard.

### `ledger` — gRPC handlers trust wire-supplied tenant_id and workload_instance_id with no application-layer SVID/identity binding; mTLS is optional and falls back to plaintext

- **Location:** `services/ledger/src/main.rs:83`
- **Category:** auth-trust
- **Impact:** Trust boundary: a caller that can reach the gRPC port can claim any tenant_id and any workload_instance_id. If it also presents a valid (or unexpired) fencing epoch for that workload it can drive reserve/commit/release for an arbitrary tenant. With mTLS unset there is zero cryptographic peer identity; even with mTLS there is no SVID-to-(tenant,workload) binding enforced. Combined with findings #1/#2 this widens the over-spend/cross-tenant blast radius.
- **Proposed fix:** Add a tonic interceptor that requires a client cert (reject plaintext in production), extracts the SPIFFE URI SAN, and asserts it matches the request's tenant_id/workload_instance_id (per-tenant SVID model already used by output_predictor per HARDEN_08). Make missing TLS an error in a production profile rather than a warn-and-continue.
- **Verifier note:** Correct in direction, feasible, and close to minimal — with two cautions. The codebase already has the exact primitives the fix needs: output_predictor/src/plugin_svid.rs provides extract_spiffe_uri_from_cert_pem() and tenant_from_subject_uri() (per HARDEN_08), so a server-side tonic interceptor that pulls the peer cert from request extensions, extracts the single URI SAN, and asserts it maps to req.tenant_id (and, where a per-workload SVID scheme exists, to fencing.workload_instance_id) is achievable without new crypto. Caution 1 (direction of the analogy): the proposed_fix says "per-tenant SVID model already used by output_predictor" — note that plugin_svid.rs is output_predictor's CLIENT-side per-tenant cert selection when it dials customer plugins (Strategy C), not an inbound server-side interceptor; so this is a reusable primitive, not a drop-in pattern. The ledger would need a NEW inbound interceptor plus a defined SVID URI scheme for sidecar/control-plane callers (the existing PREDICTOR_CLIENT prefix is wrong for this caller class). Caution 2 (fail-closed rollout): "make missing TLS an error in a production profile" is the right fail-closed move, but it must be gated on an explicit profile flag (mirror the chart.profile=production / acknowledgeNoNetworkPolicy pattern in networkpolicy.yaml:379-385) so POC/demo plaintext still works; flipping it unconditionally would break the 6 demo modes. Also bind the workload_instance_id side, not just tenant_id — binding tenant alone still lets a compromised same-tenant workload spoof another workload's fencing owner. Recommend: (a) add the inbound interceptor with an explicit ledger SVID URI scheme, (b) enforce SVID-tenant (and SVID-workload where modeled) equality, (c) gate plaintext rejection behind a production profile env flag with a loud refuse-to-start, consistent with the existing fail-closed posture.

### `output_predictor` — Per-tenant rate limiter never evicts idle buckets and fails-closed-to-DENY for all new tenants once capacity is reached — unauthenticated table-exhaustion DoS

- **Location:** `services/output_predictor/src/server.rs:327`
- **Category:** resource
- **Impact:** A caller can issue Predict with 4096 distinct fabricated tenant UUIDs (the limiter inserts a bucket for each on first sight before checking tokens) to permanently fill the map; thereafter every genuinely-new tenant's Predict RPC returns ResourceExhausted forever (until process restart). ### `stats_projector` — Project optimistic-retry loop re-issues the Signal 1 DB query on every contended retry with no iteration bound

- **Location:** `services/run_cost_projector/src/server.rs:440`
- **Category:** performance
- **Impact:** Under concurrent Project calls for the SAME run_id (e.g. sidecar retry storms), each loser re-issues a DB query per retry, multiplying DB load and risking the 5ms p99 SLO for that run. Concurrent same-run traffic is expected to be rare, and each writer eventually wins once, so livelock is unlikely — but the unbounded re-query under contention is a latent amplification.

It recommends hoisting the `signal_1_predicted_remaining_steps` call above the loop, claiming its inputs are "stable enough."
